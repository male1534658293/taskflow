# Viking First 执行规范

本规范适用于当前项目后续的分析、实现、排障与迭代。

## 核心原则

每次开始处理任务前，先判断该问题是否能依托 Viking 现有内容解决。这里的“Viking 内容”包括：

- `resources` 下的项目资料、接口文档、组件说明
- `memories` 下的偏好、实体、事件、模式
- 既有的目录结构、抽象方式、命名方式和存储逻辑

结论优先级如下：

1. 先服从 Viking 中已有的明确事实和结构
2. 再结合项目代码现状做实现决策
3. 如果 Viking 不足，再补充新的实现推导

## 每次运行的执行顺序

1. 先查 Viking 中是否已有可复用资源、记忆或模式
2. 判断当前任务属于“直接复用”“局部扩展”还是“完全新增”
3. 如果与 Viking 既有逻辑冲突，优先解释冲突点，不直接绕过
4. 只有在 Viking 无法覆盖时，才允许新增逻辑
5. 新增逻辑若被验证有效，需要补回 Viking 对应资源或记忆

## 与 aPaaS 汇总文档的重合规则

基准文档：

- `/Users/bytedance/openviking_workspace/viking/default/resources/apaas_project/apaas_api_and_pitfalls/飞书_aPaaS_平台_API_接口与踩坑记录_汇总文档.md`

后续涉及飞书 aPaaS 的任务，必须把该文档与 Viking 逻辑做重合处理，具体规则如下：

- 文档中的 SDK 用法，视为接口事实层
- 文档中的踩坑记录，视为实现约束层
- 文档中的数据对象字段，视为领域实体层
- 文档中的组件 API，视为复用模式层

处理相关任务时，不允许只看代码实现而忽略上述四层信息。

## Viking 重合映射

### 1. SDK 用法 -> 资源与事实

当任务涉及 `sdk.data`、`sdk.flow`、`sdk.func`、云函数 SDK、组件 SDK、Lark Open API 时：

- 优先从 Viking `resources` 中读取既有资料
- 将文档中的调用方式视为默认正确写法
- 新实现不得无依据偏离这些接口约束

### 2. 踩坑记录 -> 模式与事件

当任务涉及 bug、行为异常、接口不稳定、字段取值差异时：

- 优先把“踩坑记录”当成先验约束
- 先按已知坑位排查，再考虑新问题
- 若出现新坑，后续应补充到 Viking 的 `events` 或 `patterns`

### 3. 数据对象字段 -> 实体模型

当任务涉及对象结构、字段名、关系字段、选项值、用户信息时：

- 以文档中的字段定义和返回结构为主
- 在 Viking 中把这些对象视为稳定实体
- 代码内的字段处理必须和文档语义保持一致

### 4. 组件 API -> 复用策略

当任务涉及 React 组件、ViewModel、`@exposed`、`@reactive`、组件方法暴露时：

- 优先复用文档中的组件组织方式
- `props -> reactive state` 同步、方法通过 `model.` 调用等规则，视为既有模式
- 新组件行为设计应尽量贴合该模式，不另起一套

## 执行输出要求

每次相关任务执行时，至少要明确三件事：

- 本次先查了哪些 Viking 内容
- 当前实现依托了哪些现有逻辑
- 哪些部分属于新增，为什么 Viking 现有内容不足

## 不满足条件时的处理

以下情况不能直接进入编码：

- 没查 Viking 就开始实现
- 已有 aPaaS 文档覆盖，但实现未对齐文档
- 已有 Viking 逻辑可复用，但重复造轮子

遇到以上情况，应先补齐检索和对齐，再继续执行。
